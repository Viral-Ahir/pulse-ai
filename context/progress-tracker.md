- **Feature 27 — Spec Generation Flow**
  - Created `trigger/generate-spec.ts`: `generateSpecTask` (`id: "generate-spec"`, `maxDuration: 600`, `retry.maxAttempts: 1`) implemented as a `schemaTask` with a Zod-validated payload (`projectId`, `roomId`, `chatHistory[]`, `nodes[]`, `edges[]`). `chatHistory` entries are `{ role: "user" | "ai", content }`. `nodes`/`edges` use loose, `passthrough`ed schemas so we accept the full React Flow shape the canvas already emits — only `id`/`data.label`/`data.shape`/`data.color` are looked at when assembling the prompt; `source`/`target`/`data.label` are read off edges. Resolves Gemini through `createGoogleGenerativeAI` (same `GOOGLE_AI_API_KEY`/`GOOGLE_GENERATIVE_AI_API_KEY` fallback pattern as the design agent), calls `generateText({ model: google("gemini-2.5-flash"), system, prompt })` where the system prompt encodes the Markdown structure (H1 title, Overview / Components / Data Flow / Interfaces & Contracts / Non-Functional Requirements / Open Questions sections only when they apply, no preamble) and constrains references to the canvas labels. `describeGraph(nodes, edges)` produces a deterministic textual representation of the canvas (per-node `id | label | shape | color` lines + per-edge `sourceLabel → targetLabel [edgeLabel]` lines, with `(no edges)` when empty); `formatChatHistory(history)` joins entries as `User:`/`Pulse AI:` paragraphs. Progress reporting is metadata-only (no Liveblocks broadcasts) since spec generation is a per-user output, not a room-wide canvas mutation — `announce(phase, message)` calls `metadata.set("phase", phase)` and `metadata.append("messages", { phase, message, at })` for `useRealtimeRun` consumers. Empty Gemini output throws `AbortTaskRunError` so retries are skipped. Errors flow through `try/catch` → `announce("error", message)` → rethrow so the run is marked FAILED. Returns `{ runId, spec }` where `spec` is the trimmed Markdown body
  - Created `app/api/ai/spec/route.ts` (`POST`): Clerk auth via `getCurrentIdentity()` (`401`), validates the entire body through a top-level Zod `safeParse` (`400` with the Zod issues array for client debugging) — body shape mirrors the task schema (`roomId`, `chatHistory[]`, `nodes[]`, `edges[]`) but DOES NOT accept `projectId` from the client per spec scope ("Do not trust a client-supplied projectId"). The server resolves project access by `getProjectIfAccessible(roomId, identity)` (since roomId is the canonical project id in this codebase) — `403` for anyone who is not the owner or a collaborator. On success, calls `tasks.trigger<typeof generateSpecTask>("generate-spec", { projectId: project.id, roomId, chatHistory, nodes, edges })` (the server-derived `project.id` is what feeds the task payload, never the request body), persists a `TaskRun` row (`{ runId: handle.id, projectId: project.id, userId: identity.userId }`), and returns `{ runId }` with `201`. The token is NOT minted inline here (unlike `/api/ai/design`) — spec generation has a separate token endpoint to match the spec's explicit two-route contract
  - Created `app/api/ai/spec/token/route.ts` (`POST`): Clerk auth (`401`), Zod-validates `{ runId }` (`400`), looks up the `TaskRun` row via `prisma.taskRun.findUnique({ where: { runId } })` and rejects with `403` if missing OR if `taskRun.userId !== identity.userId` (defense against a different signed-in user trying to read someone else's run). Mints a Trigger.dev public access token via `triggerAuth.createPublicToken({ scopes: { read: { runs: [runId] } }, expirationTime: "1h" })` and returns `{ token }`. The 1h expiry matches the existing `/api/ai/design` pattern and gives the frontend room to reconnect without re-minting on every render. Run ownership is verified through Prisma — Trigger.dev has no per-run owner concept, so the local `TaskRun` table IS the ownership source for any future-time access
  - Build verified: `npx tsc --noEmit` is clean; `npm run build` compiles and type-checks cleanly. Both `/api/ai/spec` and `/api/ai/spec/token` register as dynamic routes in the manifest

The composer disable is keyed off `sharedActive = isActiveAiStatusPhase(phase)` so EVERY user in the room sees the same disabled composer while any user's run is in flight — not just the triggering user. The `useRealtimeRun` per-user lifecycle is preserved on top of this; the feed is a SHARED status layer, the realtime run is a PER-USER chat history layer. To make all this work, `LiveblocksProvider` + `RoomProvider` were lifted out of `CanvasWrapper` into `WorkspaceShell` so both the canvas and the AI sidebar are inside the same room — `CanvasWrapper` now only owns the suspense boundary and `useErrorListener` connection-error UI. `LiveCursors` was also updated to project the AI's `thinking` presence into the cursor name badge as a small `<Loader2 className="animate-spin" />` next to the name, so peers see when the AI (or any future thinking-aware user) is actively working.
- Canvas autosave + load (`lib/canvas-blob.ts`, `app/api/projects/[projectId]/canvas/route.ts`, `hooks/use-canvas-autosave.ts`): blob path is fixed at `canvas/{projectId}.json` with `addRandomSuffix: false` + `allowOverwrite: true` so every save replaces the previous snapshot in place — there's never a second blob to GC and the Prisma `canvasJsonPath` URL stays stable across saves. Reads bypass the SDK and just `fetch(canvasJsonPath, { cache: "no-store" })` because Vercel Blob exposes a public URL; the response shape (`{ nodes, edges }`) is validated before being returned. The `PUT`/`GET` routes both reuse `getCurrentIdentity` + `getProjectIfAccessible` (`lib/project-access.ts`) — collaborators and the owner can both save/load, anyone else gets `403`. The autosave hook is intentionally simple: it stringifies the current `{ nodes, edges }` per render, skips its very first invocation (initial hydrated state shouldn't trigger an immediate redundant write), short-circuits when the stringified payload matches the previous one, and otherwise queues a `setTimeout(1500)` that posts the JSON to the route. The load + autosave are bridged in `components/editor/canvas.tsx`: the surface gates `autosaveEnabled` behind a `hydratedRef`-guarded initial fetch so the hook can't fire until after the snapshot load attempt completes — that's the race fix for "first render with empty room overwrites a saved snapshot." The load itself is doubly gated by `nodes.length === 0 && edges.length === 0` *both* before issuing the fetch and again on the response, so a peer that joined mid-fetch is never overwritten. Status is reported via `onSaveStatusChange` up through `CanvasWrapper` → `WorkspaceShell` → `EditorNavbar`, which renders a small `bg-subtle` pill with `Loader2/Check/AlertCircle` icons; the pill only mounts when `saveStatus` is passed so editor-home navbar usage is unchanged.

- Spec persistence + download (`prisma/models/spec.prisma`, `lib/spec-blob.ts`, `trigger/generate-spec.ts`, `app/api/projects/[projectId]/specs/[specId]/download/route.ts`): the `generate-spec` Trigger.dev task now owns end-to-end persistence — it produces a `crypto.randomUUID()` specId, calls `uploadSpec(projectId, specId, spec)` to write `specs/{projectId}/{specId}.md` to Vercel Blob, and then `prisma.projectSpec.create({ data: { id: specId, projectId, filePath: blobUrl } })`. Doing the blob write before the Prisma row insert avoids orphan DB rows on a blob failure (orphan blobs on a DB failure are acceptable — they're never referenced). The task's output shape grew from `{ runId, spec }` to `{ runId, specId, spec }` so the sidebar can chain the download route off the same `useRealtimeRun` payload. The download route is `GET /api/projects/[projectId]/specs/[specId]/download` and uses the same `getCurrentIdentity` + `getProjectIfAccessible` access pattern as the canvas + chat routes — 401 → 403 → 404. The 404 branch fires both when the spec id doesn't exist and when `spec.projectId !== project.id` (defense in depth against guessing a sibling project's spec id). On success the route returns the raw Markdown with `Content-Type: text/markdown; charset=utf-8`, `Content-Disposition: attachment; filename="spec-{id}.md"`, and `Cache-Control: no-store`. `lib/spec-blob.ts` mirrors `lib/canvas-blob.ts` but does NOT use `allowOverwrite` — each spec id has its own path, so collisions are impossible and overwrites are explicitly disallowed.
- Spec generation backend (`trigger/generate-spec.ts`, `app/api/ai/spec/route.ts`, `app/api/ai/spec/token/route.ts`): `POST /api/ai/spec` and `POST /api/ai/spec/token` are split into two routes (unlike the design pipeline's single-call shape) because the spec for this feature explicitly defines both endpoints. The spec route Zod-validates the body via `safeParse` (returns `400` with the `issues` array on failure), then resolves project access by calling `getProjectIfAccessible(roomId, identity)` — `roomId` IS the canonical project id in this codebase (see Feature 07 — slug + 6-char suffix), so the route never accepts a client-supplied `projectId`. The validated `nodes`/`edges` schemas are intentionally loose (`.passthrough()` on every level) so the canvas can send the full React Flow node/edge shape without a transformation layer; the task only reads `id`/`data.label`/`data.shape`/`data.color` and `source`/`target`/`data.label`. The `generate-spec` task uses `generateText` (not `generateObject`) against `gemini-2.5-flash` because the output is free-form Markdown, not a structured graph — the system prompt constrains the section ordering (Overview / Components / Data Flow / Interfaces & Contracts / Non-Functional Requirements / Open Questions), forbids preamble, and tells the model to reference components by their canvas label rather than node id. Progress reporting is metadata-only (`metadata.set("phase", ...)` + `metadata.append("messages", ...)`) since spec generation is per-user, not a room-wide event — no Liveblocks broadcast/feed/presence chain, unlike the design agent. The token route validates `{ runId }` (`400`), looks up the `TaskRun` row via Prisma, and rejects with `403` if missing or if `userId !== identity.userId` — this is the access-control wedge: `TaskRun` IS the ownership source since Trigger.dev has no per-run owner concept. The token's `scopes.read.runs: [runId]` is narrow on purpose so even a leaked token can only read one specific run.
- AI sidebar shell (`components/editor/ai-sidebar.tsx`): a single file owning `AiSidebar` plus three internal sub-components — `AiSidebarHeader`, `AiArchitectTab` (with a nested `EmptyState`), and `SpecsTab`. The parent (`WorkspaceShell`) still controls `isOpen`/`onClose` and the slide animation is unchanged: `fixed top-12 right-0 z-40` + `translate-x-0`/`translate-x-full` + `transition-transform duration-300 ease-in-out`. Surface chrome is `bg-base/95 backdrop-blur border-l border-surface-border shadow-2xl shadow-black/40`. The sidebar is `w-96 max-w-[100vw]`, slightly wider than before, because the chat composer + message bubbles need more horizontal room than the old placeholder. Inside, a shadcn `<Tabs defaultValue="architect">` lives in a flex column (`flex-1 flex flex-col min-h-0`) so the active `TabsContent` fills the remaining height — `TabsList` is overridden to `grid grid-cols-2 w-full bg-subtle h-9` because shadcn's default `inline-flex w-fit` would shrink to content width. Active triggers use `data-active:bg-ai/15 data-active:text-ai-text` (the spec's "accent" interpreted as the AI accent, since the entire sidebar is the AI workspace and the project's shadcn `--accent` token is the cyan brand-dim color), inactive triggers stay `text-copy-muted`. Chat state is local: `messages: { id, role, content }[]` + `input: string` — there is no AI/Liveblocks/backend wiring yet, so sending only appends a user bubble. User bubbles are `self-end max-w-[85%] rounded-2xl border-2 border-brand/50 bg-brand-dim text-copy-primary`, assistant bubbles are `self-start max-w-[85%] rounded-2xl border border-surface-border bg-elevated text-ai-text`; both use `whitespace-pre-wrap break-words` so multi-line / long-word content wraps cleanly.
